## 一、 什么是 Linux？

简单来说，**Linux 是一个开源、极其稳定且高度可定制的操作系统内核。**

你可以把它理解为一个“透明且高度自由的管家”**：

- **Windows 像是一套装修好的精装房**：墙不能拆，水管不能动，如果你想在卧室里装一个车床，Windows 会告诉你“不支持此操作”。
    
- **Linux 像是一套全透明的毛坯房**：每一根电线、每一块砖头你都能看见并随意修改。如果你觉得系统图标太丑，可以改；如果你觉得内核处理数据不够快，可以自己重写驱动。
    

---

## 二、 为什么 RoboMaster 要用 Linux？

在 RM 赛场上，几乎所有的顶级视觉算法都跑在 Linux（通常是 Ubuntu）上。这不是因为 Linux 比较“酷”，而是因为**生存需求**：

### 1. 毫秒级性能与实时性 (Real-time)

在视觉对抗中，从摄像头抓拍到装甲板，到计算出云台转动角度，整个过程通常不到 **10ms**。

- **Windows** 后台会有很多无法控制的任务（如自动更新、杀毒软件），这会产生不可预测的延迟（抖动）。
    
- **Linux** 允许我们精细管理系统资源。我们可以告诉系统：“除了我的视觉算法，不准任何程序使用这个 CPU 核心。”
    

### 2. ROS 2（机器人的官方语言）

RoboMaster 现在通用的 **ROS 2** (Robot Operating System) 是一套为机器人开发的通信框架。

- 虽然 Windows 也能跑 ROS 2，但在 Linux 上的兼容性、文档支持和社区资源要好得多。
    
- Linux 的**命令行工具**（如你正在学的那些）能极大地简化机器人多个模块（摄像头、雷达、串口）的联调。
    

### 3. 硬件驱动的深度控制

RM 的视觉方案通常涉及高性能工业相机（如迈德威视、海康）和 NVIDIA 板卡（如 Jetson 或天选 5 上的显卡（天选5是主包的电脑，让他出个镜））。

- Linux 环境下，这些硬件的驱动更加轻量，且可以直接通过代码调用底层的显卡计算单元（CUDA），避免了 Windows 图形界面的额外开销。
    

### 4. 容器化（Docker）的完美契合

Docker 在 Linux 上的性能是原生的。

- 你可以把调好的环境打包成镜像。哪怕比赛现场板卡烧了，换一块新的，**30秒**就能把一模一样的开发环境部署好，这种容错率是 Windows 无法提供的。

## 三、 总结：Linux 是连接代码与物理世界的桥梁

在机器人开发中，我们追求的是**“确定性”**：

- 当我给一个指令，它必须立刻执行。
    
- 当我部署一个环境，它必须在任何地方都跑得通。
    

**这就是为什么你要忍受那些枯燥的命令和报错去学 Linux。**

学习 Linux 文件系统，最核心的一点是：**忘掉 Windows 的“分盘”逻辑（C 盘、D 盘）。**

在 Linux 的世界里，一切都是**一棵树**。

---

### 1. 核心哲学：一切皆文件

这是 Linux 最著名的准则。无论是你的代码、硬盘里的照片、鼠标、键盘，甚至是内存，在 Linux 看来统统都是**文件**。

- **好处**：你可以用同样的方式（读和写）去操作所有东西。
    

---

### 2. 树状目录结构 (The Hierarchy)

Windows 是并列的“森林”（多个盘符），而 Linux 只有**一个根**，用 `/`（斜杠）表示。

#### 必须要记住的几个“房间”：

- **`/` (Root)**：根目录，所有东西的起点。
    
- **`/bin` & `/sbin`**：放最基础的命令（比如 `ls`, `cd`）。
    
- **`/home`**：**最重要的地方**。每个用户都有自己的小窝。你的窝就在 `/home/zguss`。
    
- **`/root`**：超级管理员（Root 用户）的私人房间，普通人进不去。
    
- **`/etc`**：放各种软件的**配置文件**。以后改 ROS 设置常来这里。
    
- **`/dev`**：设备目录。你的工业相机、串口通信口（USB转串口）连上后，都会出现在这里。
    
- **`/mnt` & `/media`**：挂载点。比如你在 WSL 里看到的 `/mnt/c`，就是把 Windows 的 C 盘“挂”到了这棵树的一个枝丫上。
    

---

### 3. 路径 (Path)

- **绝对路径**：从根开始的完整地址。比如 `/home/zguss/rm_vision_task`。
    
- **相对路径**：从你现在呆的地方开始。
    
    - `.` 代表当前目录。
        
    - `..` 代表上一层目录。
        
- **`~` (波浪号)**：这是快捷键，代表你的“家” (`/home/zguss`)。
    

---

### 4. 文件权限 (Permissions) —— 视觉组的“深水坑”

你在 Linux 里经常遇到的 `Permission denied`（拒绝访问），就是因为这套机制。输入 `ls -l` 就能看到它们。

- **r (Read)**: 读。
    
- **w (Write)**: 写（修改）。
    
- **x (Execute)**: 执行（运行程序/脚本）。
    
- **三组人**：文件所有者、同组人、其他人。

wsl2大概可以理解为在windows系统下运行的一个linux子系统可以帮助我们学习linux，实际上wsl2应该可以完成大部分rm相关的操作，但是
### 为什么你最终需要实机 Linux？

虽然 WSL2 已经支持了 GPU 加速和 USB 挂载（通过 `usbipd`），但在 RM 这种追求极低延迟的场景下，它有几个致命伤：

- **工业相机兼容性：** RM 常用的工业相机（如迈德威视、大恒）往往需要特定的内核驱动。WSL2 使用的是微软定制的 Linux 内核，安装这些 SDK 时经常会遇到内核版本不匹配或无法加载模块的问题。
    
- **硬件延迟与带宽：** 工业相机通常是高带宽输入（USB 3.0 或千兆网口）。经过 WSL2 的虚拟化层后，丢帧率会升高，这在需要 100-200 FPS 的视觉识别中是不能接受的。
    
- **网络通信：** 如果你们战队使用 GigE（网口）相机，WSL2 的网络地址转换（NAT）模式会让你在配置相机 IP 和接收数据包时非常痛苦。
    
- **环境对齐：** 你们最终跑在机器人上的设备通常是 **Intel NUC** 或 **NVIDIA Jetson**，它们运行的都是原生 Ubuntu。在原生系统上开发可以保证“所见即所得”，避免“我电脑上能跑，小电脑上报错”的情况。
