# 理解 gcc/g++ 编译器

## 为什么有个 "gcc" 又有个 "g++"

这个问题我被问过很多次。

简单说：gcc 可以编译 C，g++ 可以编译 C++。但它们实际上是一套工具链的前端。

gcc 全称是 GNU Compiler Collection，原本只支持 C，后来变成了支持多语言的编译器集合。g++ 是专门针对 C++ 的调用方式，它会自动加上 C++ 标准库的链接。

所以：

```bash
gcc main.c      # 编译 C
g++ main.cpp    # 编译 C++
```

你用 g++ 编译 .c 文件也能跑，但行为可能和 gcc 略有不同（比如异常处理、运行时库的差别）。反过来的情况更麻烦：gcc 编译 .cpp 文件会报一堆关于 std:: 的错误。

**一般规则：用 C++ 就用 g++，别混。**

---

## 编译的四个阶段

很多人以为 "编译" 就是一步。但其实它分四步：

### 1. 预处理（Preprocessing）

处理 `#include`、`#define`、条件编译指令。

```bash
g++ -E main.cpp -o main.ii
```

生成的 `.ii` 文件里，你的 `#include <iostream>` 被替换成了实际的头文件内容。一个简单的 `cout` 可能膨胀成上万行代码。

### 2. 编译（Compilation）

把预处理后的代码翻译成汇编语言。

```bash
g++ -S main.ii -o main.s
```

这个阶段做语法检查、语义分析、优化。报 "syntax error" 就在这里。

### 3. 汇编（Assembly）

把汇编语言转成机器码，生成目标文件 `.o`。

```bash
g++ -c main.s -o main.o
```

这时候你拿到的是二进制文件，但还**不能执行**，因为可能引用了其他文件定义的函数。

### 4. 链接（Linking）

把多个目标文件合并，加上启动代码（crtbegin、crtend），解析符号引用，生成最终可执行文件。

```bash
g++ main.o other.o -o run
```

undefined reference 报错就在这里。

## 常用编译选项

### -I、-L、-l：头文件路径、库文件路径、链接库

这三个选项是配置 include 和 library 搜索路径的。

```bash
g++ main.cpp -I/home/meko/include -L/home/meko/lib -lopencv_world -o run
```

- `-I`：加在 include 搜索路径（`-Ipath`）
- `-L`：加在 library 搜索路径（`-Lpath`）
- `-l`：指定要链接的库（`-lopencv_world` 实际找 libopencv_world.so）

**顺序很重要！** 链接器是从左到右处理 `-l` 的。所以如果 A 库依赖 B 库，得写成 `-lA -lB`。

### -std：C++ 标准版本

```bash
g++ -std=c++17 main.cpp -o run
```

不加这个，gcc 5 之后默认用 c++14。但 RM 视觉项目可能依赖 C++17 的特性（结构化绑定、if constexpr 等），建议明确指定。

### -D：定义宏

```bash
g++ -DDEBUG main.cpp -o run
```

相当于代码里写了 `#define DEBUG`。常用于 debug 模式下开启日志：

```cpp
#ifdef DEBUG
    std::cout << "value: " << x << std::endl;
#endif
```

### -Wall、-Wextra、-Werror：警告控制

```bash
g++ -Wall -Wextra -Werror main.cpp -o run
```

- `-Wall`：打开常见警告
- `-Wextra`：打开更多警告
- `-Werror`：把警告当错误处理，编译失败

**强烈建议加 `-Wall -Wextra -Werror`**。很多 undefined reference、type mismatch 的问题编译器早就警告过了，你只是没看到。

### -O：优化级别

```bash
g++ -O0 main.cpp -o run    # 不优化，编译快，调试友好
g++ -O2 main.cpp -o run    # 平衡优化
g++ -O3 main.cpp -o run    # 更激进优化
g++ -Ofast main.cpp -o run # 忽略严格标准合规，换取极限性能
```

`-O0` 是 debug 模式常用的，因为优化会打乱代码顺序，让 gdb 调试困难。Release 发布一般用 `-O2` 或 `-O3`。

## 静态链接和动态链接

这是理解库链接的关键。

### 静态链接（Static Linking）

编译时把库的代码**复制**进最终可执行文件。

```bash
g++ main.cpp -static -lopencv_world -o run
```

生成的文件大，但**不依赖外部 .so 文件**。拷贝到任何同架构机器都能跑。

### 动态链接（Dynamic Linking）

编译时只记录"需要这个库"，运行时才加载。

```bash
g++ main.cpp -lopencv_world -o run
```

可执行文件小，但目标机器必须装了对应版本的 .so 文件。RM 上板子部署时经常遇到"找不到 .so"的问题，这时候就得 `ldd run` 查依赖，或者把 .so 拷贝进 `/lib`。

## 符号表和 name mangling

编译完的 .o 文件里有什么？

每个函数、变量都有一个**符号名**。C++ 因为支持重载，符号名会被"修饰"（name mangling）：

```
_ZNK6Eigen6MatrixIdLi3ELi1EEcvQl_v   // 这是一个 Eigen Matrix 的类型转换函数
```

如果你用 C++ 写了给 C 调用的接口，得用 `extern "C"` 阻止 mangling：

```cpp
extern "C" void detect_roi(int x, int y, int w, int h) {
    // C++ 代码，但可以被 C 代码调用
}
```

不加 `extern "C"` 的话，C 代码链接时会找不到符号，因为符号名被 mangling 成了 C++ 风格。

## CMake 里的编译器选项

直接调 g++ 的好处是直观。但项目大了，手写编译命令就疯了。

CMake 里可以这样：

```cmake
add_executable(run main.cpp)

target_compile_options(run PRIVATE
    -Wall -Wextra -Werror
    -std=c++17
    -O2
)

target_link_libraries(run opencv_world pthread)
```

这样 CMake 会把这些选项传给 g++。不同 target 可以有不同的选项，互不影响。

## RM 视觉项目里的编译器问题

板子上报错，PC 上不报错？大概率是**编译器版本或标准库版本差异**。

RM 的视觉模组可能是 ARM 架构，编译器是 aarch64-linux-gnu-g++，不是普通的 x86_64-linux-gnu-g++。

交叉编译需要指定：

```bash
aarch64-linux-gnu-g++ main.cpp -o run --sysroot=/path/to/arm/sysroot
```

有时候还涉及 softfloat/hardfloat 的 ABI 选择。选错了计算结果会错，但编译器不报错。

---

## 查编译信息

想知道 g++ 实际用了哪些路径和选项？

```bash
g++ -v main.cpp -o run
```

加上 `-v` 会输出详细的搜索路径、链接参数、驱动版本。include 找不到、library 找不到的问题，加 `-v` 基本能定位。

---

## 小结

gcc/g++ 的核心逻辑不复杂：预处理 → 编译 → 汇编 → 链接。搞懂这四步，很多错误你就知道去哪找了。

但别忘了，真正写项目的时候，没人会在终端里一行行敲 g++ 命令。那是 CMake 的工作。
